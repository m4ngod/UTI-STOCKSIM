# Requirements Document

## Introduction
本规范聚焦本期前端改动：
- 新增：市场行情面板支持“创建新标的”，并在流通股/市值/初始价格三者中，任意输入两者自动计算第三者。
- 改动：智能体面板支持“批量创建多策略散户（MultiStrategyAgent）”，可配置数量、初始资金与初始策略；同时统一前后端智能体配置/蒸馏 API 接入。
- 缺陷修复：
  1) 语言设置无效（始终显示英文）。
  2) 智能体状态显示不准确（未自动刷新）。

上述能力需与后端契约保持一致，满足性能与可观察性要求，并遵循 app 目录内的前端组织方式便于打包发布。

## Alignment with Product Vision
- 提升策略研发效率与可观察性（可视化、低延迟、可导出）。
- 降低接入成本（统一 API 契约，标准化面板与服务调用）。
- 保证可重复性（设置/布局/快照可持久与回放）。

## Requirements

### R1 市场面板：创建新标的（新增）
**User Story:** 作为量化研究员/交易员，我希望在前端快速创建新标的，填写必要参数并自动推导缺失值，从而在仿真中快速纳入测试标的。

#### Acceptance Criteria
1. WHEN 用户在“市场行情面板/标的管理”点击“创建标的” THEN 弹出对话框，包含字段：名称、代码、初始价格、总股本、流通股、市值。
2. WHEN 用户在{流通股, 市值, 初始价格}中输入任意两项 THEN 系统 SHALL 自动计算第三项：
   - 流通股 = 市值 / 初始价格（向下取整到整数股，防止小数），
   - 市值 = 流通股 × 初始价格，
   - 初始价格 = 市值 / 流通股（保留价格精度，四舍五入到最小价格步长）。
3. IF 任一输入不合法（负数、空、非数字、超范围、代码重复） THEN 系统 SHALL 给出就地错误提示并禁用“确认”按钮。
4. WHEN 用户点击“确认” THEN 系统 SHALL 调用后端创建标的接口；成功后：
   - 该标的 SHALL 出现在关注列表/可选列表中；
   - 行情面板 SHALL 能正常展示其K线/分时/成交量；
   - 事件总线 SHALL 广播“instrument-created”事件以驱动 UI 更新与日志记录。
5. WHEN 创建失败 THEN 系统 SHALL 显示错误详情并允许重试；输入保持不丢失。
6. 性能：本地校验与计算即时，提交后 UI 可见更新 ≤ 200ms（同机环境，数据可用后）。

### R2 智能体面板：批量创建多策略散户（改动）
**User Story:** 作为研究者，我希望一次性批量创建N个多策略散户智能体，统一设置初始资金与初始策略，以便进行群体行为与鲁棒性测试。

#### Acceptance Criteria
1. WHEN 用户点击“批量创建散户” THEN 弹窗包含：数量N（1–1000，默认10）、初始资金（默认10,000，≥0）、初始策略（下拉，来自可用策略清单），可选随机种子。
2. IF 任一字段非法 THEN 显示就地错误并禁用确认。
3. WHEN 点击确认 THEN 系统 SHALL 通过统一创建 API 逐个/批量请求创建 MultiStrategyAgent：
   - 成功与失败的计数与详情 SHALL 在弹窗或通知中反馈；
   - 列表与排行榜 SHALL 在创建完成或部分完成时自动刷新；
   - 创建过程 SHALL 具备进度反馈与可取消能力（最佳努力）。
4. WHEN 创建成功 THEN 新智能体 SHALL 出现在“智能体列表”与“排行榜”中，并接入行情/撮合事件流。
5. API 契约：前端 SHALL 仅通过 app.controllers.agent_creation_controller 等统一入口调用，便于后续蒸馏与新模型无缝接入。

### R3 智能体统一配置/蒸馏 API（配套）
**User Story:** 作为工程师，我希望前端通过统一 API 修改智能体配置并触发蒸馏任务，从而减少多处分支逻辑与兼容成本。

#### Acceptance Criteria
1. 提供统一方法：create/update/configure/distill(agent_id, payload)；配置变更 SHALL 版本化（记录提交人、时间与差异）。
2. WHEN 触发蒸馏 THEN 前端 SHALL 展示任务状态（排队/进行中/完成/失败）与日志摘要；完成后自动通知。
3. 安全：前端在脚本上传/配置变更前进行语法/AST 校验与黑名单规则检查（沿用 app/security）。

### B1 语言设置无效（缺陷修复）
**User Story:** 作为用户，我希望切换语言后，界面文本立即生效并在下次启动保持。

#### Acceptance Criteria
1. WHEN 用户在“设置面板”切换语言 THEN 全局 UI SHALL 在 ≤1s 内完成文案替换（无需重启）；
2. 语言偏好 SHALL 持久化于 settings_store，并在启动时优先加载；
3. i18n loader SHALL 从 app/i18n/en_US.json 与 zh_CN.json 加载；
4. 面板标题、菜单、按钮、通知与错误消息 SHALL 覆盖 ≥ 95% 的可见文本；
5. 提供单元/集成测试：切换语言后关键面板标题断言通过。

### B2 智能体状态未自动刷新（缺陷修复）
**User Story:** 作为用户，我希望智能体状态变化能被 UI 准确、及时地反映。

#### Acceptance Criteria
1. WHEN 后端推送状态变更事件 THEN 前端列表/面板 SHALL ≤500ms 内刷新对应智能体的状态（运行中/暂停/停止等）。
2. 前端 SHALL 订阅事件总线（首选）并提供退避轮询（兜底，默认2s）；
3. 状态字段（名称、类型、运行时间、最后心跳） SHALL 与后端一致且具备时间戳；
4. 提供端到端测试：模拟状态变更事件后 UI 断言更新。

## Non-Functional Requirements
### Code Architecture and Modularity
- 单一职责：面板、适配器、服务、DTO 分层清晰；
- 模块化：复用 app/ui/adapters 与 app/services，禁止直接跨层耦合；
- 契约清晰：controllers 暴露稳定接口，DTO 位于 app/core_dto；

### Performance
- UI 响应：交互到可见反馈 ≤ 200ms（本地）；
- 批量创建：100 个智能体在 10s 内完成并全部可见；
- 状态更新：P95 ≤ 500ms；

### Security
- 上传脚本 AST/黑名单校验，禁用危险内置；
- 用户输入校验与防注入；

### Reliability
- 事件驱动优先，轮询兜底；
- 失败可见、可重试，必要时部分回滚/幂等；

### Usability
- 错误就地提示与可修复建议；
- 国际化全覆盖与可回退到默认语言；
